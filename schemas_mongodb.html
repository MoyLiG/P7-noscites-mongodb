🔑 Shard Key choisie
{ city: 1, _id: 1 }
Clé composée — l'ajout de _id évite les jumbo chunks : sans lui, tous les docs Paris auraient la même clé et MongoDB ne pourrait pas fragmenter les données.
🗺️ Zone Sharding
PARIS → shardParis
LYON → shardLyon
Les zones garantissent que les données d'une ville restent toujours sur le shard dédié — même après un rebalancement automatique du cluster.
📦 Chunks
5 chunks total
Paris: 1 | Lyon: 4
Un chunk = une plage de shard key. Le balancer déplace automatiquement les chunks entre shards pour équilibrer la charge.
Exemples de routage automatique par mongos :
db.listings.find({city:"Paris"})
↓
→ shardParis uniquement (targeted query)
db.listings.find({city:"Lyon"})
↓
→ shardLyon uniquement (targeted query)
db.listings.countDocuments()
↓
→ tous les shards (scatter-gather) → merge
③ Persistance — Où sont les données après redémarrage ?

MongoDB écrit les données sur disque via WiredTiger. Les processus s'arrêtent mais les fichiers restent. start_mongodb.sh relance les processus — aucun réimport nécessaire.
